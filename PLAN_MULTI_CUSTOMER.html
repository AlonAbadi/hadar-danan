Replicate the solution to many customers

Goal Context

Sell this product (funnel + CRM + memberships + payments) to many customers. Each gets a marketing site that looks custom-made while the underlying feature engine stays the same.

Head start: the codebase already has a per-customer site generator — the Atelier deploy pipeline (app/api/admin/atelier/deploy/route.ts). It forks a template repo, injects a generated lib/client.ts config (brand, colors, copy, products, pages, feature-flag modules), creates a Vercel project, sets env vars, and deploys. It just stops at a preview demo — Step 5 injects PREVIEW_MODE=true with placeholder Supabase/Cardcom/Resend keys. The whole job is promoting that to live.

Decisions locked with you

  • Isolated instance per customer — own repo fork + own Vercel project + own Supabase project + own lib/client.ts. Not a shared multi-tenant DB.
  • Done-for-you onboarding — extend the existing Atelier admin workflow; no self-serve portal.
  • Separate payments/leads — each customer is a distinct business with its own Cardcom, Resend from-email, WhatsApp number, and data → per-customer real secrets.
  • Platform subdomain — {slug}.PLATFORM_DOMAIN attached per Vercel project (no custom-domain automation in v1).

Map Architecture

Control plane

This app (beegood.online) + its Supabase. Runs the Atelier admin, holds the customer registry, and orchestrates provisioning.

Customer instance

A fork of atelier-client-template on its own Vercel project + own Supabase + own real secrets. The custom look comes entirely from lib/client.ts + public/* assets in that fork.

Known structural risk — engine drift: every customer is a fork of the engine, so bug fixes / new features don't propagate automatically. Addressed in Phase 5.

Why Why isolated-instance, not a shared multi-tenant DB

  • Each customer collects their own leads and takes their own payments → they need their own Cardcom/Resend/WhatsApp and their own data store anyway.
  • A shared DB would force per-row tenant isolation + strict RLS for no benefit here.
  • Isolated forks make the "custom look" trivial — it's just a different client.ts.

Plan Phased implementation

v1 = minimal done-for-you provisioning. Later phases automate. Legend: NEW new file · MOD modified file.

  1. Customer registry + secrets-safe bookkeeping · this repo

    Track many live customers; capture go-live inputs without persisting secrets.

    • supabase/migrations/034_customers.sql NEW — customers table: slug (unique), subdomain, status (provisioning/preview/live/suspended/error), non-secret infra refs (supabase_project_ref, supabase_url, vercel_project_id, github_repo, production_deployment_id), checklist booleans (supabase_schema_applied, cardcom_set, resend_set, whatsapp_set, subdomain_attached), went_live_at. RLS on (service-role only). No secret columns ever.
    • app/api/admin/atelier/customers/route.ts NEW — CRUD mirroring applications/route.ts (reuse isAdminAuthorized + safeFrom).
    • Relationship: lead/onboarding funnel stays in atelier_applications; customers is the ops record created at go-live, back-referencing the application.

  2. The go-live route — real Supabase + per-customer secrets · this repo

    A "Promote to Live" action that replaces preview stubs with real infra. Keep /deploy (preview) untouched and separate.

    app/api/admin/atelier/go-live/route.ts NEW — copy scaffolding from deploy/route.ts. Body carries pasted real secrets (Supabase URL/anon/service-role, Resend key + from-email, Cardcom terminal/name/password, WhatsApp). Steps:

    1. Validate the customers row exists and is preview-deployed (has vercel_project_id).
    2. Generate CRON_SECRET/MEMBERS_SECRET/ADMIN_PASSWORD via crypto.randomUUID(); return once; never persist.
    3. Upsert Vercel env (reuse shape at deploy/route.ts:337-345 with ?upsert=true): PREVIEW_MODE=false, real Supabase keys, real Resend, real Cardcom (incl. password), NEXT_PUBLIC_APP_URL=https://{slug}.{PLATFORM_DOMAIN}, generated secrets, real WhatsApp. Prefer Vercel encrypted env type for secrets.
    4. Attach subdomain (Phase 3).
    5. Trigger fresh production deploy (env only applies on next build) — reuse deploy/route.ts:351-362.
    6. Update customers: status='live', checklist booleans, went_live_at, new production_deployment_id.

    Fresh customer DB: use the existing consolidated supabase/schema.sql (single paste) instead of replaying 33 migrations. Gate go-live on the supabase_schema_applied checkbox.

    Latent bug to fix: the current Step 5 sets CARDCOM_TERMINAL + CARDCOM_API_NAME but never CARDCOM_API_PASSWORD — live checkout fails Cardcom auth without it.
    Secret-safety: never log the request body (existing routes log only err.message — keep that); never write Cardcom/Resend/service-role values to the DB; secrets travel admin→route→Vercel over TLS only.

  3. Subdomain attach · this repo, inside go-live

    • New control-plane env PLATFORM_DOMAIN (e.g. sites.beegood.online).
    • Attach: POST /v10/projects/{id}/domains body {"name":"{slug}.{PLATFORM_DOMAIN}"}; verify via GET /v9/projects/{id}/domains/{domain} (poll for verified:true, don't hard-fail on lag).
    • One-time manual setup: wildcard *.{PLATFORM_DOMAIN} on the Vercel team + DNS CNAME * → cname.vercel-dns.com so per-customer verification is automatic.
    • Note: NEXT_PUBLIC_FROM_EMAIL uses the customer's verified Resend domain — independent of the subdomain.

  4. Admin UX integration · this repo

    • app/admin/atelier/[id]/AtelierOnboardClient.tsx MOD — Go-Live panel near handleDeploy (line 302): password inputs (never pre-filled), handleGoLive(), display one-time generated secrets. Provisioning checklist bound to the customers row.
    • app/admin/clients/page.tsx MOD (exists) — render the customers registry (slug, subdomain, status) for fleet visibility.
    • app/api/admin/atelier/orchestrate/route.ts — keep go-live human-gated in v1 (agent must not inject secrets).

  5. Engine drift fix · separate template repo — NOT this repo

    Do this AFTER live provisioning works — largest workstream, out of v1.

    • Recommended: extract the shared engine into a versioned private npm package (@beegood/engine) imported by both this repo and the template. Forks then hold only lib/client.ts + assets + a thin package.json; updates ship via npm update + redeploy.
    • v1 interim (optional): a gh script that opens PRs across all customers.github_repo to pull template main.

Test Verification — onboard one real test customer

  1. Apply migration 034 in control-plane Supabase; confirm via the migrate route probe.
  2. Preview path unchanged: run the pipeline to awaiting_approval, click Deploy, confirm {slug}.vercel.app loads in PREVIEW_MODE (checkout 503s, no DB writes).
  3. Create the customer's Supabase project; paste supabase/schema.sql; run scripts/verify-schema.mjs against it; tick supabase_schema_applied.
  4. Create the customers row from the application (POST /customers).
  5. Set up the customer's Resend (verified from-email) + a real Cardcom terminal.
  6. Go-Live: paste Supabase + Resend + Cardcom (incl. password) + WhatsApp. Route generates admin/cron/members secrets, upserts all Vercel envs (PREVIEW_MODE=false), attaches subdomain, triggers production deploy. Save the one-time admin password.
  7. Verify https://{slug}.{PLATFORM_DOMAIN}: loads with valid SSL; auth/magic-link works (real Supabase → users row in customer DB); checkout no longer 503s — complete a ₪1 test_1 purchase, webhook writes to customer DB, Resend sends a receipt from the verified from-email; WhatsApp CTA real; /admin accepts the generated password.
  8. Registry: customers row shows status='live', all checklist booleans true, went_live_at + production_deployment_id set.
  9. Secret-safety audit: grep control-plane logs + the customers row — confirm NO Cardcom/Resend/service-role values were persisted or logged.
